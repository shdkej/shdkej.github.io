## 왜 이 문서가 필요한가

노트를 쌓고, 경험을 기록하고, 위키를 퇴고하는 과정은 이미 돌아가고 있다.
빠진 건 "밖으로 내보내는 파이프라인"이다.
위키는 내 것이고, 블로그는 독자의 것이다. Threads는 그 사이의 다리다.

이 문서는 만다라트 8개 축의 지식을 콘텐츠로 변환하는 전략이다.
과설계 없이, 일상에서 자연스럽게 흘러가도록 설계한다.

---

## 콘텐츠 철학

세 줄로 요약하면:
- 인터넷에 널린 이야기 말고, 내가 겪은 이야기
- 아직 좋은 글이 별로 없는 적당한 범위의 키워드
- 경험과 감정으로 기억에 남게. 결론 먼저, 이야기로 풀기

하지 않는 것:
- AI가 생성한 글을 그대로 올리지 않는다
- 완성되지 않은 문장, 코드만 긴 글, 링크 모음은 안 한다
- 구체적인 A-Z 가이드는 다른 블로그에 있다. 나는 "선택지에서 어떤 선택을 했는지"를 쓴다

## 블로그 콘텐츠 맵

### 지식 깊이 × 희소성 매트릭스

내가 깊이 아는 것 중에서, 한국어로 양질의 글이 부족한 주제가 최고의 후보다.

```
              희소성 높음
                  │
    ★ 최우선      │    ◆ 좋은 후보
    (깊이+희소)    │    (트렌드+희소)
                  │
──────────────────┼──────────────────
                  │
    ○ 시리즈화     │    △ 선택적
    (깊이+경쟁)    │    (학습중)
                  │
              희소성 낮음

깊이 높음 ────────────────── 깊이 낮음
```

### ★ 최우선 — 깊이 있고 희소한 주제

내가 직접 겪은 실전 경험. 한국어 시장에서 구체적 수치와 의사결정 과정을 다룬 글이 드문 영역.

| 주제 | 만다라트 축 | 핵심 각도 | 상태 |
|------|------------|-----------|------|
| EKS 마이그레이션 실전기 (1.20→1.27) | 2.DevOps | 기술적 판단 + 롤백 전략 + 팀 소통 | 소재 있음 |
| Slack 배포봇으로 배포시간 70% 줄인 이야기 | 2.DevOps | 문제 인식→설계→효과 측정 | 소재 있음 |
| 분당 5만건 메시지 처리 최적화 | 2.DevOps | 병목 발견→해결→수치 변화 | 소재 있음 |
| 야간 파드 스케줄러로 AWS 비용 20% 줄이기 | 1.Infra | 비용 분석→자동화→ROI | 소재 있음 |
| MAU 300만 이커머스 모니터링 체계 설계 | 2.Monitoring | 무엇을 측정할지 결정하는 과정 | 소재 있음 |

### ◆ 좋은 후보 — 트렌드 + 희소 (지금 쓰면 선점)

AI 오케스트레이션은 아직 중급이지만, 실전 적용 사례 자체가 희소하다.

| 주제 | 만다라트 축 | 핵심 각도 | 상태 |
|------|------------|-----------|------|
| Claude Code + MCP로 업무 자동화한 실전기 | 2.AI | 도구 소개가 아닌 워크플로우 변화 | 진행중 |
| 멀티 에이전트로 PA Flow 구축하기 | 2.AI + 3.Work | 사람의 판단 + AI의 실행 | 진행중 |
| 만다라트 노트 시스템으로 지식 관리하기 | 4.Blogging | 기존 Note_Management 확장 | 업데이트 |
| 개인 인프라를 K3s로 운영하는 이야기 | 1.Infra | 비용 vs 학습 vs 편의 트레이드오프 | 소재 있음 |

### ○ 시리즈화 — 깊이 있지만 경쟁도 있는 주제

이미 좋은 글이 많지만, "내 경험"이라는 차별점으로 시리즈로 풀 수 있다.

| 주제 | 만다라트 축 | 핵심 각도 | 상태 |
|------|------------|-----------|------|
| Clean Architecture 실전 적용 일지 | 1.Architecture | 이론→실전 갭, 타협한 지점들 | 소재 있음 |
| "책처럼 읽히는 코드" 방법론 | 1.Coding | 나만의 코딩 철학 정리 | 소재 있음 |
| DevOps는 포지션이 아니라 문화다 | 2.DevOps | 조직 변화 경험 | 소재 있음 |

### 빈 영역 (아직 블로그로 안 나간 만다라트 축)

- **3.Integration**: Business, Work 관점의 경험 글 없음
- **5.Human**: 개발자의 성장/밸런스 에세이 없음
- **6.Health**: 개발자 루틴/체력관리 없음
- **7.Idea**: 여행/영화에서 얻은 개발 인사이트 없음

이 영역들은 Threads에서 먼저 반응을 테스트하고, 반응이 좋으면 블로그로 확장한다.

## Threads 루틴

### 원칙

Threads는 블로그의 축소판이 아니다. 별도의 매체다.
- 습득한 정보와 경험을 가볍게 공유하는 곳
- "꾸준히 계속 하는 사람"이라는 인상
- 자기의 표현을 하는 사람
- 향후에 궁금증을 갖게 하는 것

### 일일 파이프라인

```
[아침]                    [저녁]                     [주간]
노트 기록/퇴고 중        하루 중 인상 깊었던 것     주간 리뷰에서
→ 공유할 만한 조각 발견   → 한 문장으로 정리         반응 분석
→ 메모해둠               → /threads-review 검수     → 블로그 후보 선정
                         → 발행
```

핵심: **별도의 "콘텐츠 만드는 시간"을 만들지 않는다.** 이미 하고 있는 노트 작업과 학습 과정에서 자연스럽게 추출한다. 의욕은 낮고, 지속성은 높게.

### 콘텐츠 유형별 가이드

| 유형 | 빈도 | 예시 | 만다라트 축 |
|------|------|------|------------|
| 오늘 배운 것 | 주 3-4회 | "EKS 노드 드레인할 때 PDB 없으면 이런 일이..." | 1~2번 축 |
| 선택의 순간 | 주 1-2회 | "Lambda vs ECS, 이 상황에서는 이걸 골랐다. 이유는..." | 1~3번 축 |
| 생각 조각 | 주 1-2회 | "미니멀하게 만든다는 건 기능을 빼는 게 아니라..." | 5~8번 축 |
| 도구/발견 | 주 1회 | "이 CLI 도구 하나로 모니터링 셋업 시간이 반으로..." | 3번 축 |

### 매체별 차이 (Marketing.md 기반)

각 매체별 글은 다 달라야 한다:
- **Threads**: 짧은 인사이트, 경험 조각, 질문 던지기
- **블로그 (shdkej.com)**: 깊이 있는 경험 정리, 선택의 과정
- **네이버 블로그**: 사진 10개 + 주석 3-4줄, 담백하게
- **긱뉴스/오픈헌트**: 프로젝트 출시/업데이트 알림

## 블로그 ↔ Threads 연결 고리

```
Threads (가설)          블로그 (합)
짧은 인사이트 발행  →   반응 확인
                  →   반응 좋은 주제 발견
                  →   노트에서 소재 모음
                  →   블로그 글로 확장
                  →   블로그 링크를 Threads에 공유
                  →   댓글/반응에서 새 인사이트 발견
                  →   다시 Threads 시드로 ↩
```

이건 변증법이다. Threads에서 가설(정)을 던지고, 반응(반)을 받고, 블로그에서 종합(합)한다.

## 측정과 개선

### 주간 리뷰 (일요일, 5분)

확인하는 것:
- 이번 주 Threads 발행 수 (목표: 5-7개)
- 어떤 유형이 반응이 좋았나
- 블로그로 확장할 만한 주제가 있었나
- 만다라트 8개 축 중 어디에 치우쳐 있나

기록: 2026-03.md 월간 로그에 한 줄로 남긴다.

### 월간 리뷰 (월말, 15분)

확인하는 것:
- 블로그 글 발행 수 (목표: 월 1-2개)
- Threads→블로그 전환된 주제
- 콘텐츠 맵에서 커버된 영역 업데이트
- 다음 달 우선순위 주제 1-2개 선정

### 분기 리뷰 (분기말, 30분)

확인하는 것:
- 콘텐츠 맵 전체 업데이트
- 새로 깊어진 영역 반영
- 빈 영역 중 다음 분기에 도전할 축 선정
- 전략 자체의 수정이 필요한지 판단
